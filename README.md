# ahpc-rs

**AHP Client вҖ” Azure HTTP Proxy (Rust)**

A high-performance, asynchronous client for the Azure HTTP Proxy protocol written in Rust. It establishes an encrypted tunnel between your local machine and a remote proxy server, supporting multiple AES cipher modes and RSA-based key exchange.

## Features

- **Pure Rust** вҖ” no C/C++ dependencies; built with `tokio`, `aes`, and `rsa` crates.
- **Asynchronous I/O** вҖ” multi-threaded Tokio runtime with configurable worker count.
- **Multiple cipher modes** вҖ” AES-128/192/256 in CFB-128, OFB, and CTR modes.
- **RSA-OAEP key exchange** вҖ” secure handshake with PKCS#8 PEM public keys.
- **Optional authentication** вҖ” SHA-256 based auth key verification during handshake.
- **Graceful shutdown** вҖ” handles `Ctrl+C` for clean termination.
- **Timeout controls** вҖ” per-connection I/O timeouts (minimum 30 seconds).

The client listens on a local address (default `127.0.0.1:8089`), accepts plaintext TCP connections, encrypts them using the negotiated AES cipher, and forwards through the proxy server. Responses from the proxy are decrypted and returned to the local application. This provides a transparent encrypted tunnel for any TCP-based application.

## Supported Ciphers

| Cipher         | Code | Key Size (bytes) | Description                    |
|----------------|------|------------------|--------------------------------|
| `aes-128-cfb`  | 0x00 | 16               | AES-128 CFB-128 feedback mode  |
| `aes-128-ofb`  | 0x03 | 16               | AES-128 Output Feedback mode   |
| `aes-128-ctr`  | 0x04 | 16               | AES-128 Counter mode           |
| `aes-192-cfb`  | 0x05 | 24               | AES-192 CFB-128 feedback mode  |
| `aes-192-ofb`  | 0x08 | 24               | AES-192 Output Feedback mode   |
| `aes-192-ctr`  | 0x09 | 24               | AES-192 Counter mode           |
| `aes-256-cfb`  | 0x0A | 32               | AES-256 CFB-128 feedback mode  |
| `aes-256-ofb`  | 0x0D | 32               | AES-256 Output Feedback mode   |
| `aes-256-ctr`  | 0x0E | 32               | AES-256 Counter mode (default) |

The cipher implementations (CFB-128, OFB, CTR) are hand-written to match `mbedtls` behavior byte-for-byte, ensuring compatibility with the AHP server.

## Usage

```bash
# Run with default config file (client.json in current directory)
ahpc

# Run with a custom config file
ahpc -c /path/to/config.json
ahpc --config /path/to/config.json

# Show version
ahpc --version

# Show help
ahpc --help
```

Once running, the client will log startup information and accept local connections:

```

INFO  ahpc > server address: proxy.example.com:443
INFO  ahpc > local address: 127.0.0.1:8089
INFO  ahpc > cipher: aes-256-ctr

```

Configure your application (browser, CLI tool, etc.) to use `127.0.0.1:8089` as its HTTP/HTTPS proxy.

## Protocol Overview

### Handshake (86-byte Cipher Info Block)

When connecting to the proxy server, the client sends an RSA-encrypted cipher info block:

```
Offset  Size  Field
------  ----  -----
0       3     Magic bytes: "AHP"
3       1     Protocol version (0 = no auth, 1 = auth)
4       1     Auth version (only when pv=1)
5       1     Cipher code (e.g., 0x0E = AES-256-CTR)
6       32    SHA-256 hash of auth_key (only when pv=1)
38      16    IV (Initialization Vector, 16 bytes)
54      32    AES session key (variable length, max 32 bytes)
```

- The entire 86-byte block is encrypted with RSA-OAEP (SHA-1) using the proxy's public key.
- Once the proxy server acknowledges the handshake, the encrypted tunnel begins.
- All subsequent traffic is encrypted/decrypted using the negotiated AES cipher in the selected mode.

### Data Transfer

- **Uplink** (local вҶ’ proxy): plaintext read from local app вҶ’ AES encrypt вҶ’ send to proxy.
- **Downlink** (proxy вҶ’ local): ciphertext read from proxy вҶ’ AES decrypt вҶ’ write to local app.
- Both directions operate concurrently using tokio tasks.
- When either direction closes or times out, the other is cancelled and all sockets are shut down cleanly.

- **`main.rs`** вҖ” Parses command-line arguments via `clap`, loads the JSON config, validates it, initializes logging, creates a multi-threaded Tokio runtime, and starts the proxy listener.
